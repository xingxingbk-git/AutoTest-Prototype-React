# 自动化测试原型 HTML 到 React 迁移设计

## 1. 目标

将 `自动化测试原型html/` 下的工作台、测试用例、测试任务三套静态 HTML 原型迁移为当前 Vite + React 工程中的组件化应用。

迁移采用“视觉与核心流程还原、代码按 React 重建”的方式：

- 保留原 HTML 最终有效的视觉样式、业务数据、页面文案和核心演示交互。
- 不复制内联脚本、DOM 查询、重复渲染器和后置覆盖逻辑。
- 不恢复已经废弃的独立“执行批次”顶级模块。
- 新增、编辑、启停、执行等演示数据通过 `localStorage` 持久化。

## 2. 产品范围

### 2.1 工作台

工作台采用任务中心化设计，包含：

- 启用用例数、测试任务数、最近通过率、异常环境数。
- 运行中的测试任务。
- 最近失败记录。
- 环境健康状态。
- 新建用例、新建任务、执行任务等快捷入口。
- 系统动态。

任务编码是运行任务的详情入口。页面不再将批次 ID 或计划名称作为主要业务对象。

### 2.2 测试用例

测试用例模块包含：

- 用例库列表。
- 按名称、编码、分类、标签、数据源、任务类型和启用状态筛选。
- 分页、全选、批量启用和批量禁用。
- 新建、编辑、复制、删除和查看历史。
- 用例基本信息配置。
- PRE_TEST、TEST、POST_TEST 三阶段子项编排。
- 子项新增、编辑、启停和阶段内排序。

### 2.3 测试任务

测试任务模块包含：

- 任务列表和筛选。
- 新建、编辑、复制和删除。
- 关联用例、共享执行源、监控环境和执行策略配置。
- 执行前确认。
- 本地模拟执行进度。
- 任务详情中的配置统计、运行统计、用例实例统计和实例通过率。
- 用例实例、监控明细、统计报告和任务设置。
- 最近失败执行历史。

一次任务执行会生成一条运行记录。运行记录只作为任务详情、执行历史和统计报告的数据，不作为独立顶级模块。

## 3. 页面架构

应用采用统一 `AppShell`：

- 深色顶部品牌栏。
- 工作台、测试用例、测试任务三个顶级导航项。
- 环境与用户操作区。
- 面包屑。
- 页面内容区。
- 全局确认弹窗、消息提示和数据恢复入口。

路由设计：

- `/dashboard`：工作台。
- `/cases`：测试用例列表。
- `/cases/new`：新建用例。
- `/cases/:caseId`：编辑用例。
- `/tasks`：测试任务列表。
- `/tasks/new`：新建任务。
- `/tasks/:taskId`：测试任务详情。
- `/tasks/:taskId/edit`：编辑测试任务。

根路径跳转到 `/dashboard`。详情页刷新后应能通过路由参数恢复当前页面。

## 4. 组件边界

### 4.1 应用级组件

- `AppShell`
- `TopNavigation`
- `Breadcrumbs`
- `EnvironmentMenu`
- `ConfirmDialog`
- `ToastProvider`
- `ResetPrototypeData`

### 4.2 通用界面组件

- `PageHeader`
- `MetricStrip`
- `Panel`
- `DataTable`
- `FilterBar`
- `StatusBadge`
- `ProgressIndicator`
- `Pagination`
- `EmptyState`
- `FormField`

### 4.3 业务模块

- `features/dashboard/`
- `features/cases/`
- `features/tasks/`
- `features/environments/`
- `features/activity/`

页面组件只负责组织业务区域。数据变更、派生统计和持久化逻辑不直接堆积在页面组件中。

## 5. 数据模型与持久化

统一的 `PrototypeStore` 管理：

- `cases`
- `tasks`
- `environments`
- `taskRuns`
- `activities`
- `metadata`

数据读取顺序：

1. 检查 `localStorage` 中的存储版本。
2. 版本兼容时读取持久化数据。
3. 不存在、损坏或版本不兼容时加载预置演示数据。
4. 每次有效业务变更后写回存储。

存储内容带独立版本号，避免未来字段调整造成页面白屏。全局菜单提供“恢复预置数据”，执行前必须二次确认。

## 6. 数据联动

主要联动链路：

1. 用例新增、修改或启停后，用例库立即更新。
2. 引用该用例的任务使用同一份用例数据计算启用数和配置范围。
3. 执行任务时生成不可变的任务运行快照。
4. 本地模拟执行完成后生成用例实例结果、监控实例结果和失败记录。
5. 工作台指标、运行任务、最近失败、系统动态和任务详情从统一数据派生。

任务运行快照保留执行时的名称、用例关系、环境和策略，后续修改任务不会改写已有历史。

## 7. 交互与异常处理

### 7.1 表单

- 使用 React 受控状态。
- 必填项保存前校验。
- 无效字段在字段附近显示错误。
- 保存成功后返回对应列表或详情并显示消息。

### 7.2 危险操作

删除用例、删除任务和恢复预置数据必须显示影响范围并二次确认。

### 7.3 执行状态机

任务执行流程：

`配置确认 → 运行中 → 汇总结果 → 完成`

执行进度在本地计时器中模拟。刷新页面后，未完成记录根据存储的开始时间恢复到合理状态，不能永久卡在中间步骤。

### 7.4 演示数据

演示数据必须覆盖：

- 通过、失败、等待、未监控等状态。
- 正常与异常环境。
- 有失败记录的执行历史。
- 可展示统计报告的任务运行。

真实派生结果为空时，可使用明确标识的预置失败样本保证原型关键证据可见，但不得覆盖用户已创建的真实演示数据。

## 8. 视觉设计

视觉严格继承原 HTML 的最终有效状态：

- 深色顶部导航。
- 浅灰页面背景和白色内容面板。
- 紧凑型传统 B 端信息密度。
- 蓝色主操作。
- 绿色通过、红色失败、橙色异常、灰色未执行。
- 小圆角、轻边框、克制阴影。
- 表格优先，不将表格页面改为卡片墙。

使用项目已有的 Lucide React 图标。按钮、筛选器、表格、状态标签、分页、弹窗和空状态必须通过共享组件保持一致。

## 9. 响应式规则

- 以常见桌面工作台宽度为主要还原目标。
- 指标区域在较窄宽度下自动换行。
- 双栏内容在窄屏下变为单栏。
- 大型表格使用横向滚动，不压缩到不可读。
- 顶部导航在小宽度下保留三个模块入口，不隐藏核心导航。
- 弹窗宽度受视口约束，表单字段允许单列排列。

## 10. 验收标准

### 10.1 核心流程

1. 新建或编辑测试用例，并配置三阶段子项。
2. 新建或编辑测试任务，关联用例与环境并执行。
3. 从工作台异常记录进入对应任务详情或监控明细。
4. 刷新页面后，新增数据和当前业务结果仍然存在。
5. 恢复预置数据后，演示环境回到初始状态。

### 10.2 工程质量

- React 构建通过。
- 页面导航和详情路由刷新正常。
- 不使用原 HTML 的全局 DOM 查询和手动事件绑定。
- 不存在多套互相覆盖的渲染逻辑。
- 共享组件与业务组件边界清晰。
- 控制台无运行错误。

### 10.3 视觉质量

- 与原 HTML 逐模块截图对照。
- 检查布局、文案、表格密度、字号、间距、颜色和状态表达。
- 检查桌面宽度和窄屏宽度。
- 检查弹窗、筛选、分页、详情切换和执行状态。

## 11. 明确不在本次范围

- 真实后端接口。
- 登录、权限和多用户协作。
- 真实测试引擎或数据库连接。
- 独立的执行批次顶级模块。
- 对原型进行全新品牌或视觉风格重设计。
